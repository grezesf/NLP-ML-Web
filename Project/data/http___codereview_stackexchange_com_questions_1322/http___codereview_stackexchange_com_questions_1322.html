Optimising a Data Structure

up vote 4 down vote favorite
2

I have the following 'custom' tree data structure that I am unsure how to classify.

This concept was to be thought of as, if given ("abb", 20) and ("abc", 30), then the following would hold true:

tree['a']['b']['b'] == 20

tree['a']['b']['c'] == 30

Or, if shown as a tree diagram:

a
 \
  b
 / \
b   c

With b and c representing the values 20 and 30 respectively.

In my current implementation, I compare each branches value against the n'th character that corresponds to this depth, therefore introducing a worst case O(256) character comparison for each depth.

This is not optimal; and what I would like to change.

Obviously my implementation is a bit sketchy in how it handles the templates at the moment; a number of things could go wrong.

#include <string>
#include <vector>

template <class T>
class PrefixTree {
    std::vector<PrefixTree> m_branches;

    char m_key;
    T m_value;

    PrefixTree(char key) : m_key(key) {}
public:
    PrefixTree() : m_key(0) {}

    void add(const std::string& key, const T& value, const size_t depth = 0) {
        //printf("add(%s, %i) : m_key(%c), depth(%li)\n", key.c_str(), value, m_key, depth);

        if (key.length() < depth + 1) {
            //printf("  setting m_value to %i @ depth(%li)\n", value, depth);
            m_value = value;

            return;
        }

        //printf("  checking branches to see if appropriate branch already exists\n");
        for (auto& t : m_branches) {
            if (key[depth] == t.m_key) {
                t.add(key, value, depth + 1);
                return;
            }
        }

        //printf("  adding PrefixTree w/ key(%c)\n", key[depth]);
        m_branches.push_back(PrefixTree(key[depth]));
        m_branches.back().add(key, value, depth + 1);
    }

    T* find(const std::string& key, const size_t depth = 0) {
        //printf("find(%s) : m_key(%c), depth(%li)\n", key.c_str(), m_key, depth);

        if (key.length() < depth + 1) {
            if (key[depth - 1] == m_key) {
                return &m_value;
            }
        }

        for (auto& t : m_branches) {
            auto ret = t.find(key, depth + 1);

            if (ret != nullptr) return ret;
        }

        // nothing found
        return nullptr;
    }
};

#include <iostream>

int main() {
   PrefixTree<int> pt;
   pt.add("aab", 20);
   auto v = pt.find("aab");
   std::cout << v << ' ' << *v << std::endl;

   return 0;
}

Update: updated algorithm, added encapsulation, removed references so could return null if nothing found.

Update 2: implemented some of the suggestions as made by Paulo Pinto, however I feel that this is now just a dense vs sparse array question.

Some suggestions:

You are using a vector for your branches but you are using an index loop to iterate through the vector.
I'd consider using the vector iterator.
This will make your code a bit clearer and versatile (if you decide to change the container type for example). Will also get rid of the double branches[i] access in each loop.

Should the n_trees be public? You could write a get function but I'm guessing the client code shouldn't be allowed to change that.

For the return value at the end, if you cast it to V, I think that might work for all types, although I'm not sure about that.

Why do you have m_parent? I don't think there are any reasonable operations on a trie that require a parent pointer in each node, and removing it can reduce the space used a lot.

I don't love using a depth argument in add. I'd use the following instead:

void add(const std::string& key, const T& value) {
  return add(key.data(), key.size(), value);
}
void add(const char *key, const T& value) {
  return add(key, std::strlen(key), value);
}
void (const char* key, std::size_t len, const T& value);  // impl below

I suggest that you keep the children sorted by m_key. Write a comparison functor like so:

struct KeyLessThan {
  bool operator()(const PrefixTree& l, const PrefixTree& r) const {
    return l.m_key < r.m_key;
  }
  bool operator()(const PrefixTree& tree, char c) const {
    return l.m_key < c;
  }
};

Now you can rewrite add like this (using the signature above):

void PrefixTree::add(const char* key, std::size_t len, const T& value) {
  assert(len >= 0);
  if (len == 0) {
    m_value = value;
    return;
  }
  auto it = std::lower_bound(begin(m_branches), end(m_branches),
                             *key, KeyLessThan());
  if (it == end || it->m_key != *key) {
    it = m_branches.emplace(it, *key);
  }
  it->add(key + 1, len - 1, value);
}

The size of m_branches is bounded to be quite small (256 elements in theory; in practice if you're doing English words only it'll be 26 or 52 at the root and much smaller in interior nodes), so this should be quite fast, and it'll make find faster.

Speaking of m_branches, m_children would be a more usual name for it.

It's not really clear that you want find to return a T*; it's odd to give out handles to internal memory in a data structure like this. I would make the signature one of

const T* find(const std::string& key) const
std::pair<bool, const T&> find(const std::string& key) const

or

std::pair<bool, T> find(const std::string& key) const

(if you have access to a C++14 implementation, even better is 

std::optional<T> find(const std::string& key) const

Whichever you choose, use the same overloads as with add for const char *, etc.

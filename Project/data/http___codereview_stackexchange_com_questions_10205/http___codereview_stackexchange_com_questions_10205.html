Trying to do a one liner to convert two vector<T> into a vector< pair<T, T> >. I decided to use transform with make_pair as the binary operator. I thought this was awesome, but to my surprise, slight changes in the binary operator function change broken code to/from working code.

When compiling with gcc 4.5.3 using c++11 I encounter this error message in my code:

gnu/4.5.1/../../../../include/c++/4.5.1/bits/stl_algo.h:4726:2: 
error: invalid initialization of reference of type 'Point_<float>&&' 
  from expression of type 'Point_<float>'

But, compiling the same code WITHOUT using the c++11 standard works fine. Could someone explain what's going on here?

As a fix for c++11 compiling, by replacing

make_pair<Vector2f, Vector2f>

with

make_pair<Vector2f&, Vector2f&>

the code compiles in std=c++0x; and I was told it has to do with c++11 rvalues and handling template parameters. That's an interesting problem!

First of all, let me simplify things a bit. You don't really need std::transform, nor your class has to be a template class to reproduce the behavior. I've boiled it down to:

#include <vector>
#include <algorithm>

struct Point2f
{
    float x, y;
};

int main()
{
    std::vector<Point2f> in1(1);
    std::vector<Point2f> in2(1);
    std::make_pair<Point2f, Point2f>(in1[0], in2[0]); //(*)
}

Which indeed results in an error in (*) line. One may start yelling "make_pair is broken", but that's not true. On the contrary, it is very well thought!

First of all, let's try to fix this example by changing the (*) line to:

std::make_pair(in1[0], in2[0]);

Apparently, template argument deduction does its job as the program is well-formed now. So how does it work?

make_pair template declaration is template <class T1, class T2> pair<V1, V2> make_pair(T1&& x, T2&& y) noexcept, which means that explicit specifying T1=Point2f, T2=Point2f, as in the original case, leads to make_pair(Point2f&& x, Point2f&&) declaration.

At the same time, both vector's subscript operator (in boiled down example) and iterator dereference (as in the guts of std::transform in your original example) return an lvalue reference to its value type: Point2f&. lvalue cannot be bound to rvalue reference, what makes the program ill-formed.

In the fixed case, the automatic deduction follows the rule from C++ Standard 14.8.2.1/3: "If (function template parameter type) is an rvalue reference to a cv-unqualiﬁed template parameter [check: T1&&] and the argument is an lvalue [check: operator[] result], the type “lvalue reference to A” is used in place of A for type deduction". In short, that means nothing else than T1=Point2f&, T2=Point2f&, which you have found to be working. :)

One last thing to discuss is what really happens in the explicit specialization w.r.t. rvalue reference to lvalue reference. The mechanical substitution of template parameters would result in make_pair((Point2f&)&& x, (Point2f&)&& y) (parentheses added for clarity). The stalemate is broken by yet another Standard rule 8.3.2/6: "If a type template-parameter denotes a type TR that is a reference to a type T, an attempt to create the type “rvalue reference to cv TR” creates the type TR.". In plain English: (Point2f&)&& -> Point2f&.

Long story short, what you want is:

std::transform(in1.begin(), in1.end(), in2.begin(),
    std::back_inserter(paired_out),
    std::make_pair<Point2f&, Point2f&>);

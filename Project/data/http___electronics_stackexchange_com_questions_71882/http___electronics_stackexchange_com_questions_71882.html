Burst mode data errors on I2C

up vote 1 down vote favorite

I'm using burst mode to read 14 bytes from a sensor via I2C running at 400KHz. The problem is that there are periodic errors on the last byte of the packet, that come in roughly every 8 reads. The issue does seem to be specifically related to the last byte, as a current workaround is to simply request 15 bytes instead of 14 (there are registers after the ones i care about, so that works fine).

What could possibly cause this? A couple of things that I'm uncertain about:

  • the pull-up resistors are 4.7K. I know this is on the threshold of sizing for 400KHz, but switching to smaller resistors doesn't seem to have an impact (tried 3.3, 2.5 and 1.5)

  • There are what appear to be irregularities in the clock line (see attached logic analyzer screenshot). I'm not sure if that's something or not, but I would expect a clock line to be... well - regular.

I'm using the hardware I2C driver on an STM32 chip.

LA output:

i2c noise stm32
